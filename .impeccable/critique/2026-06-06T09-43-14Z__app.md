## Anti-Patterns Verdict

LLM assessment (independent): NOT AI slop. A Linear/Notion/Stripe-fluent user would trust it. Positive craft: a committed warm identity (not violet-default), hand-desaturated event fills (`oklch(from … l calc(c*0.85) h)`), three non-color event-status treatments (cancelled stripes, planned dotted outline, inactive wash), and a contrast/token system with measured AAA ratios baked into the comments. The dense real week stays scannable.

Deterministic scan: 2 findings, both the `border-l-4` side-tab on task cards (`task-card.tsx:74`, `task-backlog-rail.tsx:74`) — unchanged this session. On a task card the left stripe carries the task's color (a defensible categorization affordance, though the parent skill bans the pattern; a leading dot or full tint would be more on-brand).

Reconciled false positive: the independent review flagged a floating "1 Issue" pill and a dark puck as debug chrome reducing polish — those are the Next.js Dev Tools overlay (dev-only, absent in production), not product UI.

## Overall Impression

The session did exactly what it set out to: the two mobile P1s, the calendar-legibility P2s, and the AAA contrast/touch-target gaps are resolved, and the fresh review independently confirms all of them as strengths now. The next tier is deeper and more interesting: the product's two headline promises — "see both people's days at a glance" and "trust that private stays private" — are undercut on the densest surface, because owner and privacy are encoded by color alone there. That, plus discoverability and error-recovery, is where the real points now live.

## What's Working (confirmed by the independent review)

1. Non-color status encoding is real and disciplined: cancelled (stripes + grayscale + strikethrough), planned (dotted inset), inactive (faint wash), overdue (struck CalendarX2 + weight + sr-only). Most calendars fail "never by color alone"; this passes because each state changes shape/texture.
2. The contrast/token system is engineered for AAA, not eyeballed: measured-ratio comments per muted value, theme-aware ink via `toPaletteInk`, re-tinting that can't silently break contrast across the five palettes.
3. Density without clutter on the Week grid: desaturated fills, recessive sleep, hairline grid, z-raise-on-hover keep a punishing real week scannable. The chrome recedes exactly as the brand brief asks.

## Priority Issues (the new tier)

- **[P1] Owner + privacy are color-only on the grid.** `event-block.tsx` shows a `Users` icon for shared but no lock for private and no owner glyph; `resolveOccurrenceColor` lets a category/custom color override the owner color, so "mine vs partner's vs private" is not answerable at a glance, and a Private item looks identical to a Visible one on your own calendar. This contradicts both the AAA "no color alone for member identity" requirement and the privacy-legibility principle, on the most-used surface. Fix: a persistent lock glyph for private + an owner cue (initial or border treatment) that survives recoloring. Command: `/impeccable harden` (or `colorize` for the identity system).
- **[P1] Hidden power: zero discoverability.** Shift+M, Ctrl+Alt+arrows, Ctrl+0, Alt/Shift-drag, bulk Delete are unsurfaced. Fix: a `?` shortcuts overlay + tooltips on toolbar buttons (the zoom-reset already shows "(Ctrl+0)" — extend that). Command: `/impeccable onboard` (+ `clarify`).
- **[P1] Developer-worded error shown to users.** `calendar-canvas.tsx:126` and `tasks-shell.tsx:154`: "Couldn't load your … Make sure the database schema is applied and seeded." Fix: human message + Retry; log the schema hint to console only. Command: `/impeccable clarify` (+ `harden`).
- **[P1] No keyboard model on the time grid.** `EventBlock` is a non-focusable `<div>`; selection/move/resize are pointer-only (only Agenda rows are keyboard-activable). PRODUCT.md requires keyboard-completable primary flows; moving an event isn't. Command: `/impeccable harden`.
- **[P2] Destructive actions aren't reversible and under-differentiate the nuclear option.** Event/task delete says "can't be undone"; the recurrence prompt's This / Following / All buttons are three same-shaped full-width rows. Fix: optimistic delete + Undo toast (matching category delete), separate/space "All events", add an occurrence count. Command: `/impeccable harden`.
- **[P2] Short blocks clip their title on a redundant time line.** `event-block.tsx` always renders the time range; sub-30-min events become unreadable slivers in week/day. Fix: height-aware layout that drops the time line below a px threshold. Command: `/impeccable layout`.

## Persona Red Flags

Alex (power user): the deep flexibility is locked away (undiscoverable shortcuts); no global saved/synced status to trust a fast edit landed.

Sam (accessibility): well served on contrast (measured AAA), sr-only overdue, 44px coarse targets — but the time grid has no keyboard nav (can't move/resize events keyboard-only), and member/privacy identity is color-only.

Casey (mobile): Agenda default + 44px targets + bottom sheets are good; rotated vertical context labels are near-unreadable one-handed, and the silent Week→3-day / auto-Agenda coercion can feel like the view "changed itself."

The Private Partner (derived): adoption hinges on trusting private stays private, yet privacy state is invisible on the grid (a Private event looks like a Visible one on your own calendar), and filing under a Shared context silently overrides the Private/Visible toggle — the one interaction most worth making legible for a trust-critical product.

## Minor Observations

- Settings is a long single-column scroll with five large pickers; the one screen drifting toward the "prosumer overload" anti-reference.
- "Context" remains non-standard wording for what most call calendars/categories.
- Date input is locale-shaped (DD/MM) without an explicit cue for a bilingual household.
- The `border-l-4` task-card stripe (detector) is the lone remaining slop-adjacent pattern.

## Questions to Consider

- If color is correctly banned as the sole signal everywhere else, why does the densest, most-used surface encode owner and privacy only in color? What would the grid look like with identity + privacy as first-class glyphs, freeing the color budget for category meaning?
- You built remarkable quiet power and then hid all of it. Is "discoverable without clutter" achievable with zero surfaced hints, or does it still need one quiet `?` doorway?
- Should destruction always be reversible (undo toast) and the confirm dialog be the thing you remove, rather than confirm-then-permanent?
- Is the privacy model's three axes (private / visible / shared, plus context override) the real job, or is the household's mental model just "ours vs mine"?
