## Anti-Patterns Verdict

**Does this look AI-generated? No.** This is a coherent, restrained product, not slop.

**LLM assessment:** Component vocabulary is consistent across week/day/3-day/month/agenda; color is rigorously *meaningful* (per-item → category → owner cascade in `resolveOccurrenceColor`) rather than decorative; ownership is encoded in fill *style* (filled = mine/shared, outlined wash = partner's) so it survives colorblindness and Catppuccin re-tinting; status uses redundant encoding (cancelled = stripes + strikethrough, planned = dotted). The shared absolute bans are clean: no gradient text, no glassmorphism, no hero-metric cards, no identical card grids, no side-stripe borders (the `1.5px` is a full ring). Two small strangeness flags: the `⟳ series` drag badge reads as debug chrome, and the sidebar uses uppercase tracked section labels ("MY CALENDAR" / "OTHER CALENDARS" / "CONTEXTS") plus uppercase gutter labels — the one "eyebrow" tell present, though defensible as sidebar group headers.

**Deterministic scan:** `detect.mjs --json` over all 14 calendar markup files returned `[]`, exit 0 — **clean, zero findings**. The intentional calendar patterns (inline `backgroundColor` event fills, `border-[1.5px]`, `z-[var(--evt-z)]` overlap cascade, `position:absolute` blocks) did not trip the detector, so there are no false positives to discount. The two assessments agree: there is no surface-level slop here.

**Visual overlays:** Not available — no dev server was running and one was not started (to avoid the project's known build/dev `.next` conflict). No live browser overlay was produced; findings are source-grounded.

## Overall Impression

The craft is real and the restraint is earned — this already reads like a tool a Fantastical/Notion user would trust on sight. The single biggest opportunity is not visual at all: **the product's stated core job is "avoid double-booking," and nothing in any view, preview, or detail panel ever acknowledges an overlap.** Fix the gap between what the product promises (coordination, clash-free) and what the calendar actually signals, and 29 becomes a confident mid-30s.

## What's Working

1. **Color stays meaningful, never decorative.** `resolveOccurrenceColor` + `eventFillStyle` make hue always answer "whose / what category," and encode ownership in fill style, not color alone — the DESIGN.md "color carries meaning" rule actually enforced in code.
2. **Status is never color-only.** Cancelled = diagonal stripes + line-through, planned = dotted outline, inactive = grayscale+opacity. Textbook redundant encoding.
3. **Recurrence scope prompt respects real mental models.** "This / This and following / All events" with the destructive "All" styled red is exactly the Google/Fantastical pattern a fluent user expects — earned familiarity.

## Priority Issues

**[P0] No double-booking / clash signal anywhere.** The core job is "avoid double-booking," yet a source-wide search finds no conflict/overlap warning; `overlaps`/`enclosingContext` in `lib/calendar/contexts.ts` serve context-nesting only. You can drop a new event squarely onto your partner's block and the app says nothing.
- *Why it matters:* the one task the product exists for has no closure; the "did I just double-book us?" anxiety is never resolved.
- *Fix:* reuse the `packDay` overlap geometry you already compute to surface a calm hairline clash marker on overlapping blocks + a one-line "Overlaps [name]'s [title]" note in `EventDetails` and the drag-create preview. Keep it whisper-weight to honor the warm-restraint personality — no red banner.
- *Command:* **/impeccable harden**

**[P1] Plain delete is instant and unconfirmed — and silently mutates your partner's calendar.** In `onDeleteEvent` (calendar-shell.tsx), a non-recurring, non-context event calls `mutations.remove(ev.id)` immediately. For a *shared* event this also removes it from the partner's view with no confirm and no visible undo.
- *Why it matters:* in a shared-calendar-for-two built on trust, a write that changes the other person's calendar with zero reassurance is the highest-stakes valley in the app.
- *Fix:* route every delete through a confirm or a persistent toast with a real Undo button; escalate copy when shared ("This also removes it from [partner]'s calendar").
- *Command:* **/impeccable harden**

**[P1] No keyboard or screen-reader path to create or move a timed event.** TimeGrid create/move/resize/multi-select all live in `onPointerDown/Move/Up`; the timed `EventBlock` is a `div` with no `tabIndex`/`role`/key handler, so it is unreachable and unannounced. Only Agenda rows and Month items are keyboard-activatable.
- *Why it matters:* this is a WCAG-AA blocker, and the project explicitly targets AA. An entire user class (keyboard-only / screen-reader) cannot use the primary week/day views at all.
- *Fix:* add a keyboard creation/move path (focusable blocks with arrow-key nudge + a "new event at selected time" command) and proper roles/labels; Agenda can be the accessible fallback but week/day shouldn't be pointer-only.
- *Command:* **/impeccable audit** (map the full a11y gap), then **/impeccable harden**

**[P1] The load-error state is a developer message with no recovery.** `CalendarCanvas`'s error branch renders "Couldn't load your calendar. Make sure the database schema is applied and seeded." to end users, near no recovery, with no Retry — and neighbor swipe-panes can show it full-screen mid-gesture.
- *Why it matters:* exposes internals, offers no way forward, breaks the calm.
- *Fix:* plain copy ("We couldn't load your calendar") + a Retry button that re-runs the window query; keep the schema hint dev-only.
- *Command:* **/impeccable clarify**

**[P2] The entire power-gesture vocabulary is undiscoverable, and two decision points exceed 4 options.** Ctrl-drag=copy, Alt=series, Shift=multi-select, Ctrl+wheel=zoom, swipe=page are surfaced nowhere; the 5-way ViewSwitcher and the 5-branch editable-event context menu both pass the ≤4 working-memory limit.
- *Why it matters:* the deepest, best interactions are invisible; a fluent user expects a `?` cheat-sheet.
- *Fix:* add a `?` keyboard/gesture sheet + contextual modifier hints once a block is grabbed; collapse Day/3-day/Week into one range control with a density toggle and demote "Assign to context" out of the menu's top level.
- *Command:* **/impeccable onboard**

## Persona Red Flags

**Alex (power user):** No Esc-to-cancel an in-flight drag (a mis-drag must be completed then undone); no "n"/"c" create shortcut or arrow-key navigation between events; the 5-way view switcher has no number-key shortcuts.

**Sam (accessibility):** TimeGrid is pointer-only — no keyboard path to create/move/resize a timed event; the timed `EventBlock` is a non-focusable `div` (no `tabIndex`/`role`), unreachable and unannounced by a screen reader; drag feedback glyphs (`⟳ series`, copy `+`) are `aria-hidden`, so even if reachable, copy-vs-move-vs-series is never announced. Whose-is-whose and shared markers are otherwise strong.

**The coordinating partner (project persona):** Whose-is-whose (outlined = partner's, filled = mine/shared, owner color, "[name]'s calendar" row) and what's-shared (Users glyph, Shared badge) are both excellent. But **spotting a clash is impossible** — two overlapping blocks just share columns via `packDay`, visually identical to two unrelated events. For the one persona the product exists to serve, the single most important signal is absent.

## Minor Observations

- `now-line` red and the `evt-cancelled` stripe red both ride the `destructive` token — current-time vs not-happening are semantically different; mild meaning collision.
- Month day-cell: clicking empty space *creates* while clicking the day *number* navigates — two actions in one small target, easy to mis-tap on touch.
- Tiny typographic drift: committed block time uses `–`, the drag preview uses `A – B` (spaced).
- The `secondaryHourLabel` / `zoneAbbrev` gutters add two uppercase micro-columns that crowd the already-narrow week grid.
- "+N more" month Popover lists items but offers no "view this day" affordance.

## Questions to Consider

1. If the north star is "avoid double-booking," what is the *calmest possible* clash signal that still honors warm-restraint — a hairline, a whisper-weight note, never a red banner?
2. Is the calendar quietly two products — a deep desktop tool and a shallow phone/AT viewer — and is that split intended or an accessibility gap you'd be uncomfortable shipping?
3. Should any write that changes your partner's calendar (delete/move a shared event) ever happen without a beat of reassurance?
